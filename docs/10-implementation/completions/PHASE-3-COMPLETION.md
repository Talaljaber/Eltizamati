## 3. RLS policies

Every table's SELECT/INSERT/UPDATE/DELETE policies match `database-schema.md` §4 exactly: simple `user_id = auth.uid()` ownership checks, no joins, no subqueries. Tables without an UPDATE or DELETE policy (`rate_periods` — append-only; `consent_records`/`profiles` — no client deletion; `calculation_runs`/`consent_records` — no client update) deny that operation to every role, including the owner, matching the frozen design's stated intent (append-only history, immutable calculation runs, service-role-only account deletion). As of the grants migration (§9 addendum), these tables are granted the operation at the table level but denied by RLS's default-deny when no policy matches — the correct pattern, verified by pgTAP.

## 4. Composite ownership foreign keys (§1.11)

`obligations` carries `UNIQUE (id, user_id)`. Every child table with a required `obligation_id` (`loan_details`, `murabaha_details`, `card_details`, `rate_periods`, `payments`) declares **only** a composite FK `(obligation_id, user_id) → obligations (id, user_id)` — no redundant single-column FK, since the composite target already implies `obligation_id` must reference a real `obligations.id`. Tables with a **nullable** `obligation_id` (`calculation_runs`, `insights`) declare both the composite FK (checked only when `obligation_id` is set — Postgres `MATCH SIMPLE` skips it otherwise) and a direct `user_id → auth.users(id)` FK (always checked, covering the aggregate/obligation-agnostic case). This is exactly the design frozen in the readiness pass, implemented verbatim.

## 9. Addendum — bugs found and fixed during real Docker-based verification (2026-07-11)

With Docker available for the first time, every previously-static-only artifact was run for real. Three genuine defects surfaced that manual review had missed:

1. **`supabase/migrations/20260712000008_calculation_runs.sql`** — the index `calculation_runs_obligation_formula_created_idx` referenced a column `created_at`, which does not exist on this table (the actual timestamp column is `calculated_at`, confirmed against `database-schema.md` §1.8). `supabase db reset` failed immediately on this migration. **Fixed:** the index now references `calculated_at`.

2. **Missing table-level `GRANT`s for the `authenticated` role.** RLS policies restrict _which rows_ a query can see, but Postgres separately requires table-level privileges (`GRANT SELECT/INSERT/UPDATE/DELETE`) before RLS is even evaluated — none of the original 10 table migrations included any `GRANT` statement. Every table was completely unreachable by the app's actual runtime role. This was invisible to the previous session because the two pgTAP files that passed (`20_ownership_and_constraints.sql`, `30_account_deletion.sql`) both run entirely as the `postgres` superuser, which bypasses both RLS and grants — only `10_rls_cross_user.sql`, which correctly switches to `set local role authenticated`, hit the real error (`permission denied for table profiles`). **Fixed:** new migration `20260712000011_grants.sql` grants all four operations on every table to `authenticated`, deliberately including operations with no matching RLS policy (e.g. `rate_periods` delete) — this makes RLS's default-deny return zero affected rows rather than a hard permission error, which is both what the pgTAP suite asserts and the correct behavior for the app.

3. **`supabase/tests/database/10_rls_cross_user.sql`** — 20 assertions used the pattern `select is((with u as (update ... returning 1) select count(*) from u), 0, '...')`. Postgres rejects data-modifying CTEs nested inside a subquery — they must be the top-level statement. **Fixed:** each assertion rewritten as `with u as (update ...) select is((select count(*) from u), 0, '...')`, putting the `WITH` at the top level of the statement.

A fourth, unrelated gap was fixed to get `pnpm check` genuinely green with the newly-generated types file present: `.claude/` (gitignored but not previously in `.prettierignore`) and the generated `database.types.ts` (Supabase boilerplate, not hand-written — lint rules for application code don't apply) were added to the relevant ignore files.

None of these four fixes touch already-reviewed business logic, RLS policy intent, or the frozen schema design — they are bug fixes to make the existing design actually run, not redesigns.
