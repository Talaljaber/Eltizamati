# Seed & Demo Data Architecture

**Home:** `packages/demo-data` — a versioned, pure-TS factory (no JSON blobs scattered in the app). The canonical demo *content* is defined in `docs/01-requirements/mvp-scope.md §4`; this file defines the mechanics.

## 1. Design rules

| Rule | Why |
|------|-----|
| Seed is generated by typed builders (`buildDemoLoan()`, …) sharing validation with real entry paths | Seed can never drift from schema; changing the domain breaks the seed loudly at compile time |
| Seed carries `seedVersion` (semver) stamped into provenance `sourceReference` | Reset/reload (FR-SET-005) and "which seed produced this bug" debuggability |
| **Date anchoring:** seed dates are computed relative to an anchor ("demo date") = install date, not hardcoded | The demo never rots: "next payment in 6 days" stays true in every rehearsal and on judging day (BR-STAT-002 edge) |
| Every derived figure in the seed is covered by TV-30x/TV-5xx vectors | Judges can challenge any number; we answer with a passing test |
| Seed values chosen for hand-verifiability (round numbers, clean splits) | Finance teammates verify with a calculator |
| Demo provider is a real `ObligationDataProvider` running the real import pipeline | The demo rehearses the production path (provider-abstraction.md §3) |

## 2. Seed contents (v1)
1. **Variable-rate personal loan** (the star): 20,000 JOD, 84m, 7.5%→9.25% at month 15 (installment unchanged), 30 months elapsed, all payments logged on time → residual insight fires, scenario TV-304 restores trajectory.
2. **Murabaha auto financing:** cost 15,000 + profit 3,600 = 18,600 JOD, 60m, 22 payments of 310 JOD → progress 36.67%, INV-7 exact math.
3. **Credit card:** limit 4,000, balance 2,350 (58.75% → renders 59%), min 3% floor 10 JOD, APR 24%, statement seeded → utilization insight at threshold 70%? *No* — seed sits below 70% deliberately so the card is calm and the loan owns the demo tension; a "raise utilization" dev toggle exists for testing the insight.
4. **Insights pre-seeded:** RATE_INCREASED (read), INSTALLMENT_UNCHANGED_AFTER_INCREASE (unread), RESIDUAL_RISK (unread) — the dashboard opens with exactly the story we want to tell.

## 3. Test fixtures
The same builders power unit/integration tests (`packages/demo-data/src/fixtures.ts`): `aLoan({overrides})`, `aPayment(...)` — one fixture vocabulary everywhere (anti-pattern guard: divergent hand-rolled test data).

## 4. Transition to real data
Demo → manual: user leaves demo mode → prompt: keep exploring vs "start fresh with my data" (erase + manual entry). Demo and personal data never mix (`dataMode` global — a demo obligation cannot coexist with a real one; simplest honest model). P1 providers onboard through the same ImportService with real provenance — no code path changes.
